# Технічне завдання: інтеграція 1С (OData / веб-публікація) із зовнішньою системою обліку товарів

**Напрямок обміну**: лише читання з 1С (запис у 1С не передбачається).

## 1. Рішення замовника (вихідні вимоги)

### 1.1. Мета першого етапу

1. Синхронізація довідника номенклатури
2. Отримання залишків по складах
3. Журнал руху товару на основі документів 1С

### 1.2. Склад даних етапу 1

- **Номенклатура**: найменування, коди, одиниці виміру, групи/папки
- **Склади**: коди, назви, ознаки (за можливістю)
- **Контрагенти**: назва, ЄДРПОУ, адреса
- **Залишки**: як у поточному Excel-звіті; за потреби — ціна за одиницю, заводський номер обладнання (якщо є в 1С)
- **Рух товару**: надходження, реалізація, переміщення, повернення, інвентаризація, списання; дата, відповідальна особа

### 1.3. Правила відбору документів

- В обмін — лише проведені документи
- Чернетки — не потрібні
- Скасовані / анульовані — не вивантажувати
- Орієнтир оновлення — не рідше ніж раз на годину

### 1.4. Склади: зіставлення та виключення

**Основна модель (зовнішня система):**

1. З 1С по OData отримується довідник складів (див. п. 1.2).
2. На стороні зовнішньою системою обліку. У «Управлінні складами» для кожного складу зовнішньої системи адміністратор обирає відповідний склад 1С зі списку; після збереження використовується Ref/GUID як жорстка прив’язка.
3. На стороні зовнішньою системою обліку. У БД зберігаються мінімум ідентифікатор складу в 1С (Ref/GUID); за потреби — код, найменування з 1С для відображення та аудиту. Запити залишків і руху виконуються для складів, у яких прив’язка задана (якщо інше не затверджено окремо).

**На боці 1С:**

- За потреби продуктивності/безпеки — узгодити підмножину складів у OData або правила відбору; інакше — повний довідник, фільтрація за прив’язками на боці зовнішньої системи.
- Опційно — налаштовуване обмеження вивантаження залишків/руху по переліку складів та по періоду дати наприклад останні 12 місяців.

### 1.5. Інфраструктура та безпека

- Бекенд зовнішньої системи — власний сервер.
- Контур доступу (HTTPS, білий список IP, VPN тощо) та тип сертифіката — за рішенням системного адміністратора.

## 2. Завдання для програмістів 1С

1. Публікація OData (за потреби — HTTP-сервіс) для об’єктів з п. 1.2.
2. Відбір лише проведених документів; без чернеток і без скасованих/анульованих.
3. Залишки та рух — з полями згідно п. 1.2; ціна за одиницю та заводський номер — якщо дані є в обліку.
4. Для руху: тип операції, дата, відповідальна особа, номенклатура, склад(и), кількість, унікальний id документа.
5. Фільтр дозволених складів для вивантаження — якщо технічно реалізовано в конфігурації (узгодити з таблицею з п. 1.4).
6. Технічний користувач з правами лише на об’єкти інтеграції.
7. Надати інформацію — документація: URL, сутності, приклади запитів, опис полів.

## 3. Завдання для системного адміністратора

1. IIS (або погоджений веб-сервер) + HTTPS, публікація бази 1С.
2. Надати інформацію — доступ з мережі сервера згідно обраного контуру.
3. Автентифікація та передати інформацію правил підключення.
4. Сертифікат; при CA — ланцюжок для довіри на боці замовника за потреби.
5. Надати інформацію: URL, облікові дані інтеграційного користувача.

## 4. Результат

- URL OData (і/або HTTP-сервісів)
- Облікові дані технічного користувача
- Опис сутностей, приклади запитів, поля для номенклатури, складів, контрагентів, залишків, руху
- Інструкція з фільтром складів (якщо реалізовано)

## Примітки по поточній кодовій базі (DTS Service)

- У бекенді вже є модель **складів** (`Warehouse`) і API `/api/warehouses`.
- Є “1С-стиль” дерево **категорій номенклатури** (`Category`) + “карточки продукту” (`ProductCard`) як канонічний довідник номенклатури.
- Зараз **залишки** підтягуються через імпорт Excel звіту 1С: `POST /api/equipment/import-stock-xlsx` (`backend/lib/stockXlsxImport`).
- Є **журнал складських документів** у MongoDB (надходження/переміщення/відвантаження/інвентаризація) з API:
  - `GET /api/documents/receipt`
  - `GET /api/documents/movement`
  - `GET /api/documents/shipment`
  - `GET /api/documents/inventory`

Наступний крок інтеграції з 1С по OData — визначити відповідність між сутностями 1С і нашими моделями/ендпоїнтами, та реалізувати періодичне оновлення (не рідше 1 разу на годину) з фільтрами “лише проведені” та “лише прив’язані склади”.
