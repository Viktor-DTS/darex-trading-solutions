# Специфікація HTTP‑сервісу 1С для інтеграції (Stage 1, read-only)

**Мета**: надати зовнішній системі стабільний HTTP API для читання довідників і даних складу/руху з 1С у форматі “як потрібно бізнесу”, без прямої роботи клієнтів із 1С.

**Напрямок обміну**: лише читання (запис у 1С не потрібен).

## 0) Загальні вимоги

- **Транспорт**: HTTPS.
- **Доступ**: з сервера зовнішньої системи (білий список IP / VPN — за рішенням адміна).
- **Автентифікація**: технічний користувач з правами *лише* на об’єкти інтеграції.
- **Формат**: JSON `application/json; charset=utf-8`.
- **Час/таймзона**: дати/час — ISO‑8601 (рекомендовано `YYYY-MM-DDTHH:mm:ssZ` або з офсетом).
- **Пагінація**: для великих довідників бажано підтримати `limit` + `offset` або `page` + `pageSize`.
- **Ідемпотентність**: всі `GET` без побічних ефектів.

## 1) Довідники (мінімум Stage 1)

### 1.1 `GET /warehouses` — довідник складів 1С

**Що робить**: повертає список складів, доступних для інтеграційного користувача.

**Навіщо потрібен**:
- щоб адміністратор зовнішньої системи зробив **прив’язку “наш склад ↔ склад 1С”** по `ref/guid`.
- щоб надалі запити залишків/руху виконувались лише для прив’язаних складів.

**Запит (рекомендовано)**:
- `?includeInactive=true|false` (за замовчуванням `false`)

**Відповідь 200**:
- масив складів
- кожен склад має **стабільний ключ** `ref` (GUID/Ref 1С)

**Поля елемента**:
- `ref` (string, required): GUID/Ref складу 1С (основний ключ)
- `code` (string, optional): код складу
- `name` (string, required): назва складу
- `flags` (object, optional): ознаки/статуси (якщо є в конфігурації), напр.:
  - `active` (boolean)
  - `deletedMark` (boolean)
  - `type` (string) — якщо є класифікація (транзитний/основний/…)

**Приклад**:

```json
[
  {
    "ref": "b3f0b6c1-6c49-11ee-8c99-00155d0a0b01",
    "code": "WH-001",
    "name": "Склад Київ (Основний)",
    "flags": { "active": true, "deletedMark": false }
  }
]
```

### 1.2 `GET /nomenclature` — довідник номенклатури (товари + групи/папки)

**Що робить**: повертає номенклатуру 1С (як позиції, так і групи/папки) для синхронізації довідника у зовнішню систему.

**Навіщо потрібен**:
- щоб у зовнішній системі був довідник позицій (найменування/код/од.виміру)
- щоб побудувати дерево груп/папок “як у 1С”
- щоб коректно прив’язувати залишки/рух до позицій номенклатури через `ref`

**Запит (рекомендовано)**:
- `?changedSince=2026-01-01T00:00:00Z` (опційно, інкремент)
- `?includeGroups=true|false` (за замовчуванням `true`)
- `?includeInactive=true|false` (за замовчуванням `false`)
- пагінація: `limit`, `offset`

**Поля елемента**:
- `ref` (string, required): GUID/Ref номенклатури 1С
- `code` (string, optional): код
- `name` (string, required): найменування
- `unit` (string, optional): одиниця виміру (напр. `шт.`)
- `isGroup` (boolean, required): `true` якщо це група/папка
- `parentRef` (string|null, optional): GUID/Ref батьківської групи (для дерева)
- `flags` (object, optional):
  - `active` (boolean)
  - `deletedMark` (boolean)

**Приклад**:

```json
{
  "items": [
    {
      "ref": "a1b2c3d4-0000-0000-0000-000000000001",
      "code": "GR-01",
      "name": "Товари",
      "isGroup": true,
      "parentRef": null,
      "flags": { "active": true, "deletedMark": false }
    },
    {
      "ref": "a1b2c3d4-0000-0000-0000-000000000010",
      "code": "PR-0001",
      "name": "Перетворювач частоти 7.5кВт",
      "unit": "шт.",
      "isGroup": false,
      "parentRef": "a1b2c3d4-0000-0000-0000-000000000001",
      "flags": { "active": true, "deletedMark": false }
    }
  ],
  "paging": { "limit": 1000, "offset": 0, "total": 12034 }
}
```

### 1.3 `GET /contractors` — довідник контрагентів

**Що робить**: повертає довідник контрагентів з 1С.

**Навіщо потрібен**:
- для довідника/підстановок у зовнішній системі
- для зв’язків у рухах (надходження/реалізація/повернення) — якщо в документах є контрагент

**Запит (рекомендовано)**:
- `?search=...` (пошук за назвою/ЄДРПОУ)
- `?changedSince=...` (опційно)
- `?includeInactive=true|false` (за замовчуванням `false`)
- пагінація: `limit`, `offset`

**Поля елемента**:
- `ref` (string, required): GUID/Ref контрагента 1С
- `name` (string, required): назва
- `edrpou` (string, optional): ЄДРПОУ / ІПН (як ведеться)
- `address` (string, optional): адреса (можна одним рядком)
- `flags` (object, optional):
  - `active` (boolean)
  - `deletedMark` (boolean)

**Приклад**:

```json
[
  {
    "ref": "c0ffee00-1111-2222-3333-444444444444",
    "name": "ТОВ \"Приклад\"",
    "edrpou": "12345678",
    "address": "м. Київ, вул. Прикладна, 1",
    "flags": { "active": true, "deletedMark": false }
  }
]
```

## 2) Дані складу та руху (Stage 1)

### 2.1 `GET /stocks` — залишки по складах (аналог Excel‑звіту)

**Що робить**: повертає залишки номенклатури по складах у форматі, який відповідає поточному Excel‑звіту з 1С (або максимально наближений до нього).

**Навіщо потрібен**:
- щоб зовнішня система показувала залишки “як у 1С”
- щоб синхронізація виконувалась не рідше ніж раз на годину

**Важливо**:
- залишки повертаються **лише по складах, які дозволені/прив’язані** (за потреби — фільтр на боці 1С)
- якщо технічно складно — сервіс може приймати перелік складів і фільтрувати в запиті

**Запит (рекомендовано)**:
- `warehouseRef=GUID` (опційно; якщо не передано — всі дозволені склади)
- `warehouseRefs=GUID,GUID,...` (опційно; перелік)
- `asOf=2026-04-30T00:00:00Z` (опційно; “на дату/час”, якщо підтримується)
- `changedSince=...` (опційно; якщо 1С може віддавати інкремент по змінах залишків)
- пагінація: `limit`, `offset` (якщо віддається великий набір)

**Поля рядка залишку** (мінімум):
- `warehouseRef` (string, required): GUID/Ref складу 1С
- `warehouseName` (string, optional): назва складу (для відображення/аудиту)
- `nomenclatureRef` (string, required): GUID/Ref номенклатури 1С
- `nomenclatureCode` (string, optional)
- `nomenclatureName` (string, required)
- `unit` (string, optional)
- `quantity` (number, required): кількість (доступний/фактичний залишок — узгодити)

**Опційні поля** (якщо дані є в обліку):
- `price` (number|null): ціна за одиницю
- `amount` (number|null): сума (якщо рахується)
- `serialNumber` (string|null): заводський/серійний номер (якщо залишки ведуться по серіях)
- `batch` (string|null): партія/серія (якщо застосовується)

**Приклад**:

```json
{
  "asOf": "2026-04-30T07:00:00Z",
  "items": [
    {
      "warehouseRef": "b3f0b6c1-6c49-11ee-8c99-00155d0a0b01",
      "warehouseName": "Склад Київ (Основний)",
      "nomenclatureRef": "a1b2c3d4-0000-0000-0000-000000000010",
      "nomenclatureCode": "PR-0001",
      "nomenclatureName": "Перетворювач частоти 7.5кВт",
      "unit": "шт.",
      "quantity": 12,
      "price": 15300
    }
  ],
  "paging": { "limit": 1000, "offset": 0, "total": 54120 }
}
```

### 2.2 `GET /movements` — журнал руху товару (по документах 1С)

**Що робить**: повертає журнал руху товарів (рядки документів) на основі документів 1С за період, з фільтрами по складах.

**Обов’язкові правила відбору (Stage 1)**:
- вивантажувати **лише проведені** документи
- **не** вивантажувати чернетки
- **не** вивантажувати скасовані/анульовані/помічені на видалення

**Типи операцій (мінімум)**:
- `receipt` (надходження)
- `sale` (реалізація)
- `move` (переміщення)
- `return` (повернення)
- `inventory` (інвентаризація)
- `writeoff` (списання)

**Запит (рекомендовано)**:
- `from=2026-01-01T00:00:00Z` (required): початок періоду
- `to=2026-04-30T23:59:59Z` (optional): кінець періоду (за замовчуванням “до зараз”)
- `warehouseRef=GUID` (optional): фільтр по складу (рухи, що зачіпають склад)
- `warehouseRefs=GUID,GUID,...` (optional): кілька складів
- `limit`, `offset` або `page`, `pageSize` (recommended)

**Формат даних**:
- Повертаються **рядки руху** (line items) — зручно для аналітики та імпорту.
- Кожен рядок має містити:
  - **ідентифікацію документа** (унікальний id + номер + дата)
  - **тип операції**
  - **відповідальну особу**
  - **номенклатуру**
  - **склад(и)** (джерело/призначення або один склад — залежно від типу)
  - **кількість** (знаком: “+” на прихід, “-” на витрату — узгодити)

**Поля рядка руху** (мінімум):
- `movementId` (string, required): унікальний id рядка (стабільний). Напр. `${docRef}:${lineNo}`
- `docRef` (string, required): GUID/Ref документа 1С
- `docNumber` (string, required): номер документа
- `docDate` (string, required): ISO‑8601 дата/час документа
- `operationType` (string, required): один з типів вище
- `responsibleName` (string, optional): відповідальна особа
- `responsibleRef` (string, optional): GUID/Ref користувача/співробітника (якщо є)
- `nomenclatureRef` (string, required)
- `nomenclatureName` (string, required)
- `unit` (string, optional)
- `quantity` (number, required)
- `fromWarehouseRef` (string|null, optional): склад‑джерело
- `toWarehouseRef` (string|null, optional): склад‑призначення
- `warehouseRef` (string|null, optional): якщо документ “один склад” (наприклад списання/інвентаризація)

**Опційні поля**:
- `contractorRef`, `contractorName` (для receipt/sale/return)
- `price` (number|null), `amount` (number|null)
- `serialNumber` / `batch` (якщо рух ведеться по серіях/партіях)
- `comment` (string|null)

**Приклад**:

```json
{
  "from": "2026-04-01T00:00:00Z",
  "to": "2026-04-30T23:59:59Z",
  "items": [
    {
      "movementId": "d0d0d0d0-aaaa-bbbb-cccc-111111111111:1",
      "docRef": "d0d0d0d0-aaaa-bbbb-cccc-111111111111",
      "docNumber": "ПР-0000123",

      "unit": "шт.",
      "quantity": 2,
      "warehouseRef": "b3f0b6c1-6c49-11ee-8c99-00155d0a0b01",
      "contractorName": "ТОВ \"Приклад\""
    },
    {
      "movementId": "e1e1e1e1-aaaa-bbbb-cccc-222222222222:1",
      "docRef": "e1e1e1e1-aaaa-bbbb-cccc-222222222222",
```

## 3) Помилки та коди відповідей (рекомендовано)

- `200` — успіх
- `400` — некоректні параметри
- `401` — неавторизовано (невірні облікові дані)
- `403` — заборонено (немає прав)
- `429` — занадто багато запитів (rate limit), якщо потрібно
- `500` — внутрішня помилка 1С‑сервісу

Формат помилки (рекомендовано):

```json
{ "error": "Human readable message", "code": "SOME_CODE" }
```

## 4) Нотатки для узгодження (важливо)

- **Єдиний ключ інтеграції**: для всіх довідників потрібен стабільний `ref` (GUID/Ref 1С).
- **Помічені на видалення / неактивні**: бажано або не віддавати за замовчуванням, або позначати прапорцями.
- **Інкрементальне оновлення**: якщо можливо, підтримати `changedSince` (дуже зменшує навантаження).
